# 「钱都去哪了」V3 完整产品版设计说明

**日期：** 2026-07-13  
**产品：** 钱都去哪了  
**作者：** tphu  
**交付形态：** 可直接分享的单 HTML 完整离线版

## 1. 背景与目标

现有版本已经具备微信、支付宝、银行账单的本地解析、跨来源去重、基础统计、隐私隔离和单 HTML 构建能力，但分析维度偏基础，页面仍有信息层级、移动端密度和视觉统一性上的提升空间。

V3 的目标是将工具升级为一个可信、易懂、可执行的个人消费洞察产品，同时保留无需安装、无需账号、无需服务器、账单不离开设备的核心优势。

成功标准：

1. 保持现有微信、支付宝、银行解析与跨来源去重能力稳定。
2. 完成消费画像、时间规律、月度变化、预算和专业建议能力。
3. 支持导出结构化 Excel 分析报告。
4. 桌面端和手机端均无重叠、横向溢出或不可操作控件。
5. 最终生成一个不依赖网络资源的离线 HTML 文件，可直接发送给朋友。
6. 全部现有测试继续通过，并为新增分析、导出和界面结构补充测试。

## 2. 设计原则

### 2.1 先保证可信，再增加丰富度

- 不重写已经稳定的解析与去重核心。
- 新功能通过独立模块扩展，避免把界面状态、统计口径和文件解析重新耦合。
- 所有画像与建议都必须能回溯到明确的交易数量、金额和时间范围。
- 样本不足时展示“数据不足”，不生成武断结论。

### 2.2 本地优先与最小留存

- 文件解析、统计、图表和导出全部在浏览器本地完成。
- 不发送网络请求，离线版继续使用 `connect-src 'none'`。
- 预算仅保存在 `sessionStorage`，关闭标签页或浏览器会话后自然清除。
- 不保存账单内容、商家明细或用户身份信息。

### 2.3 先给结论，再允许追溯

- 首页先回答“这个月花了多少、比上月怎样、钱主要去了哪里”。
- 第二层解释变化原因和消费习惯。
- 第三层提供可执行建议、数据质量和交易明细。
- 所有图表均提供文本或表格替代信息，关键含义不只依赖颜色。

## 3. 方案选择

采用“稳定内核 + 增量模块”方案。

保留：

- `src/core.js` 中现有归一化、分类、跨来源去重与基础分析接口。
- `src/app.js` 中文件读取、PDF Worker 配置和顺序解析策略。
- 现有本地第三方依赖和单 HTML 构建模式。

新增：

- `src/insights.js`：消费画像、时间分布、月度环比和可解释建议。
- `src/budget.js`：预算状态、会话级存储和预算对比。
- `src/exporter.js`：Excel 工作簿生成。
- 对 `src/app.js` 做界面编排级改造，不复制分析规则。

不采用全面重写，因为它会增加支付宝表头兼容、PDF 表格还原和跨来源去重回归风险；也不采用只换皮方案，因为无法满足升级方案中的专业分析目标。

## 4. 数据处理架构

```text
文件选择
  → 类型与大小校验
  → 按文件顺序解析
  → 各平台字段归一化
  → 无效状态过滤与质量警告
  → 跨来源去重和关联
  → 统一分类与资金流口径
  → 基础财务分析
  → 消费画像 / 时间规律 / 月度变化 / 预算对比
  → 页面渲染和 Excel 导出
```

### 4.1 解析阶段

- 微信：兼容官方 XLS/XLSX 导出，保留交易状态、退款和交易单号处理。
- 支付宝：兼容 CSV 编码、引号、逗号、不同换行符和常见表头别名。
- 银行：继续支持横向与旋转 PDF 表格、加密 PDF 密码和支付渠道商户还原。
- 多文件始终顺序处理，避免手机同时解压或解析大文件造成内存峰值。

### 4.2 财务口径

- 净消费 = 消费支出 - 退款。
- 投资、理财申购、内部转账和债务偿还不计入消费。
- 结余 = 真实收入 - 净消费。
- 结余率仅在真实收入大于零时计算，否则显示“无法计算”。
- 同来源只有明确相同交易单号才去重；跨来源仅在日期、金额、渠道与时间候选唯一时关联。

## 5. 新增分析能力

### 5.1 月度变化

- 最近 6 个月收入、净消费、结余和结余率趋势。
- 当前月对上月的金额变化和百分比变化。
- 分类变化贡献排行，用于解释“为什么这个月多花或少花”。
- 不对不完整月份做未经说明的直接比较；界面会提示覆盖天数。

### 5.2 消费画像

- 工作日与周末：金额、笔数、客单价和主要分类。
- 时段分布：早晨、午间、下午、傍晚、深夜。
- 星期 × 时段消费热力图。
- 高频商家：次数、总额、平均金额、最近消费日期。
- 餐饮场景：早餐、午餐、晚餐、外卖、咖啡与饮品。
- 习惯标签只有在满足最小样本数和时间跨度时出现，并同时展示依据。

### 5.3 专业建议

每条建议包含：

- 观察：具体发生了什么。
- 依据：金额、笔数、时间范围或环比变化。
- 行动：一个清晰且可执行的下一步。
- 影响：可估算时给出保守金额范围，否则不虚构收益。

禁止内容：

- 不推荐具体理财产品、借贷产品或信用额度。
- 不使用羞辱性、焦虑式或绝对化语言。
- 不根据少量样本给用户贴性格或生活方式标签。

### 5.4 预算

- 支持设置月度总预算和分类预算。
- 展示已使用金额、剩余金额、预算进度和按当前节奏的月底预测。
- 金额输入使用数字键盘友好配置并进行非负数校验。
- 预算只写入当前会话，不进入离线构建文件，也不与账单一起导出。

### 5.5 Excel 导出

导出一个 `.xlsx` 工作簿，包含六张表：

1. `总览`：范围、收入、净消费、结余、结余率、预算状态。
2. `月度趋势`：每月收入、净消费、退款、结余和环比。
3. `分类分析`：分类金额、占比、笔数、客单价、必要/可选属性。
4. `商家分析`：商家次数、总额、均值、最近日期和主要分类。
5. `交易明细`：统一后的交易字段、来源、规则、置信度与匹配信息。
6. `数据质量`：读取数、忽略数、重复关联数、歧义数和警告详情。

为降低敏感信息扩散风险，导出前明确提示 Excel 会包含交易明细，由用户主动确认。

## 6. 界面与交互设计

### 6.1 视觉方向

采用“清新、自然、可信赖”的轻量数据产品风格：

- 主色使用低饱和薄荷绿，搭配清透中性色和少量状态色。
- 通过留白、细边界和柔和层级组织内容，减少厚重阴影与过多卡片。
- 标题明确但不过度装饰，金额使用稳定的等宽数字特征。
- 支付宝和微信入口继续使用官方视觉可识别的本地图标。
- 支持浅色与深色主题，图表、边框、文字和状态色均使用主题变量。

### 6.2 页面结构

1. 顶部导航：产品名、隐私说明、主题、Excel 导出、导入账单。
2. 导入阶段：三平台入口、拖拽区域、文件限制、解析进度和示例数据。
3. 总览：净消费、收入、结余率、预算进度、月度走势、分类排行。
4. 变化：环比结论、变化贡献和本月重点关注。
5. 消费画像：习惯标签、热力图、商家排行和餐饮场景。
6. 建议：按重要程度展示有依据的行动建议。
7. 明细与质量：筛选、交易表、来源统计和解析警告。

### 6.3 响应式行为

- 以手机宽度为默认布局，再增强至平板和桌面。
- 320–430px：所有主内容单列，按钮可换行或占满可用宽度。
- 431–767px：指标两列，其余核心图表单列。
- 768–1199px：主次两列，根据图表最小可读宽度调整。
- 1200px 以上：总览保持稳定内容宽度，不无限拉长折线图和明细文字。
- 所有触控目标至少 44 × 44px；正文在手机上不小于 16px。
- 明细表在小屏改为摘要行与展开详情，不直接压缩成不可读宽表。

### 6.4 状态与反馈

- 导入中显示当前文件、总进度和正在执行的阶段。
- 单个文件失败时保留其他文件的成功结果，并给出可操作的错误原因。
- 没有数据、样本不足、没有预算和没有可比月份分别使用独立空状态。
- 图表渲染失败时仍先显示文字结论和交易明细。
- 动效仅用于 150–300ms 的状态切换，遵循 `prefers-reduced-motion`。

## 7. 错误处理与边界情况

- 文件扩展名与真实内容不一致：拒绝读取并提示重新导出。
- 文件过大或总大小超限：在解析前阻止，避免手机崩溃。
- 加密 PDF：只在当前内存使用密码，不记录、不回显。
- 未识别表头：展示识别到的列名和对应平台重新导出指引。
- 部分退款：生成独立退款调整记录，不修改原交易金额。
- 重复候选不唯一：全部保留并写入数据质量警告。
- 月份只有少量天数：标注为不完整月份，不生成误导性环比结论。
- 零收入月份：不展示伪造的 0% 结余率。
- 没有时间字段：仍参与金额统计，但不进入时段画像。
- 离线浏览器不允许 PDF Worker：给出可理解的兼容提示，不影响 CSV/XLSX 解析。

## 8. 构建与交付

- 源码仍采用可维护的多文件结构。
- `scripts/build-offline.js` 按固定顺序内嵌 CSS、第三方库、PDF Worker 和业务模块。
- 构建产物固定为 `output/钱都去哪了-离线版.html`。
- 产物不包含外部脚本、样式、图片或网络地址，不包含测试账单和个人资料。
- 分享说明保持诚实：电脑浏览器兼容性最好；手机需先保存文件后用支持本地 HTML 的浏览器或文件工具打开，iPhone/iPad 对本地脚本的支持取决于具体打开方式。

## 9. 测试与验收

### 9.1 自动化测试

- 保留当前 46 项测试作为回归基线。
- 为消费画像的样本阈值、时段边界、工作日/周末和商家聚合增加单元测试。
- 为环比、不完整月份和变化贡献增加单元测试。
- 为预算校验、会话存储和预测增加单元测试。
- 为 Excel 六张表、字段顺序、公式注入防护和空数据增加测试。
- 更新结构测试，校验新增模块被普通构建和离线构建完整包含。

### 9.2 浏览器验收

- 使用示例数据验证完整导入、分析、预算和 Excel 导出流程。
- 在 375、768、1024、1440px 四种宽度检查布局。
- 检查浅色、深色、高对比度和减少动态效果设置。
- 检查键盘操作、焦点可见性、表单标签和图表文本替代。
- 使用包含支付宝 CSV、微信 XLSX 和银行 PDF 的组合数据做一次真实回归。

### 9.3 离线验收

- 构建产物只有一个 HTML 文件。
- 断网后可完成 CSV、XLSX 和 PDF 解析。
- 产物中不存在外链资源、用户路径、真实账单或身份信息。
- 重新运行构建结果稳定，测试全部通过。

## 10. 非目标

本次不实现：

- 云端同步、账号体系、跨设备数据同步。
- 后端服务或数据库。
- 自动连接支付宝、微信或银行账号。
- 投资收益预测、征信评估或贷款建议。
- Electron、原生手机应用或应用商店发布。
- 依赖新在线 CDN 的运行时功能。

这些边界确保项目继续保持“一个文件即可分享、账单只在本地”的核心定位。
